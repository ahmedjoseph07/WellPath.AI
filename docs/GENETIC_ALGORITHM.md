# Genetic Algorithm — Well Path Optimization

> **Thesis context:** After XGBoost classifies each depth interval, WellPath.AI uses a Genetic Algorithm (GA) implemented with the DEAP framework to find the optimal directional well trajectory that maximises productive zone exposure while minimising dogleg severity.

---

## 1. Why Optimization Is Needed

XGBoost tells us **where** the productive zones are (depth intervals with high productivity scores). But it does not tell us **how to drill through them**. The well path must:

1. Start at the surface (0, 0) vertically
2. Deviate through productive intervals
3. Reach a target total depth
4. Not exceed safe dogleg severity limits (sharp bends damage casing and drill string)
5. Maximise contact with productive rock

This is a **complex combinatorial optimisation problem**:
- The decision variables are the inclination and azimuth at each survey station
- The objective is non-linear (interpolated productivity along a 3D curve)
- There is no closed-form gradient → classical gradient descent fails
- Multiple local optima exist

**Genetic Algorithms** are ideal for this class of problem.

---

## 2. Biological Analogy

GAs are inspired by **natural selection** (Darwin, 1859):

| Biology | Genetic Algorithm |
|---|---|
| Individual organism | Candidate well trajectory (solution) |
| Chromosome / DNA | Array of inclination/azimuth angles |
| Gene | Single inclination or azimuth value |
| Population | Set of candidate trajectories evaluated in parallel |
| Fitness | Objective function value (how good the trajectory is) |
| Natural selection | Selection of better-performing individuals |
| Crossover / mating | Combining two parent trajectories → offspring |
| Mutation | Random perturbation of angles |
| Generation | One round of selection + crossover + mutation |
| Evolution | Iterative improvement over generations |

## 3. Problem Encoding

### 3.1 Decision variables

A well trajectory is defined by **survey stations** — points along the borehole at which inclination (Inc) and azimuth (Az) are measured. WellPath.AI uses $n_w$ waypoints.

**Chromosome structure:**

```
Individual = [Inc₁, Az₁,  Inc₂, Az₂,  Inc₃, Az₃, ..., Inc_{n-1}, Az_{n-1}]
              └─ gene 0 ─┘  └─ gene 2 ─┘  ...
```

- Length: $2(n_w - 1)$ genes (first station is fixed vertical: Inc=0, Az=0)
- $n_w = 8$ waypoints by default → 14 genes per individual
- Inc bounded: [0°, 85°] — avoid horizontal/reverse drilling
- Az bounded: [0°, 360°]

### 3.2 Decoding an individual

```python
def decode_individual(individual, n_waypoints, start_depth, end_depth):
    md_array  = linspace(start_depth, end_depth, n_waypoints)
    inc_array = [0.0]   # first station: vertical
    az_array  = [0.0]   # first station: north
    for k in range(n_waypoints - 1):
        inc_array.append(individual[2*k])
        az_array.append(individual[2*k + 1])
    return md_array, inc_array, az_array
```

The decoded arrays are passed to the **Minimum Curvature Method** to compute 3D Cartesian coordinates (see `DIRECTIONAL_DRILLING.md`).

## 4. Population Initialization

The initial population is generated by sampling each gene **uniformly at random** within its bounds:

```python
def init_individual():
    genes = []
    for _ in range(n_waypoints - 1):
        genes.append(random.uniform(0.0, 85.0))   # inclination
        genes.append(random.uniform(0.0, 360.0))  # azimuth
    return Individual(genes)

population = [init_individual() for _ in range(population_size)]
```

Default population size: **50 individuals** per generation.

A diverse initial population is critical — it ensures the search starts from many different starting points in the solution space, reducing the chance of converging to a poor local optimum.

## 5. Fitness Function

The fitness of a trajectory is a **weighted combination of two objectives**:

$$F = \underbrace{P_{\text{exposure}}}_{\text{maximise}} - \underbrace{w_{DLS} \cdot DLS_{\text{penalty}}}_{\text{minimise}}$$

where:

### 5.1 Productive zone exposure

$$P_{\text{exposure}} = \frac{1}{N_{pts}} \sum_{j=1}^{N_{pts}} s\bigl(z_j\bigr)$$

- $z_j$ = TVD at each trajectory point $j$
- $s(z)$ = productivity score at depth $z$, linearly interpolated from the XGBoost output
- Average productivity along the trajectory — ranges from 0 to 1

### 5.2 Dogleg severity penalty

$$DLS = \frac{\Delta\theta}{\Delta MD} \times 30$$  (in degrees per 30 m)

where $\Delta\theta$ is the total angle change between consecutive survey stations computed using the **Minimum Curvature tangential averaging** formula.

The penalty is applied when DLS exceeds a soft threshold to prevent mechanically infeasible trajectories.

### 5.3 Weighting

The `dls_weight` parameter (default 0.3) controls the trade-off:

| `dls_weight` | Behaviour |
|---|---|
| 0.0 | Pure zone-exposure maximisation; ignores bends |
| 0.3 | Balanced — default |
| 1.0 | Very conservative — strongly penalises high DLS |

## 6. Selection — Tournament Selection

WellPath.AI uses **tournament selection** (tournsize = 3):

```
For each slot in the new generation:
  1. Pick 3 individuals randomly from the population
  2. Choose the one with the highest fitness
  3. Place it in the mating pool
```

**Why tournament selection?**
- Simple and efficient — O(n) per generation
- Tournament size controls **selection pressure**: larger tournament → stronger pressure toward top performers
- Does not require fitness scaling (unlike roulette wheel selection)

---

## 7. Crossover — Simulated Binary Bounded (SBX)

WellPath.AI uses **SBX crossover** (`tools.cxSimulatedBinaryBounded`), which is the standard for real-coded GAs.

### How SBX works

For each gene position $k$, SBX computes a spread factor $\beta$ from a polynomial distribution:

$$\beta_q = \begin{cases}
(2u)^{1/(\eta+1)} & \text{if } u \leq 0.5 \\
\left(\frac{1}{2(1-u)}\right)^{1/(\eta+1)} & \text{if } u > 0.5
\end{cases}$$

where $u \sim \mathcal{U}[0,1]$ and $\eta = 20$ is the **distribution index** (higher $\eta$ = offspring closer to parents).

The offspring genes are:

$$c_1^{(k)} = \frac{1}{2}\left[(1+\beta_q) p_1^{(k)} + (1-\beta_q) p_2^{(k)}\right]$$
$$c_2^{(k)} = \frac{1}{2}\left[(1-\beta_q) p_1^{(k)} + (1+\beta_q) p_2^{(k)}\right]$$

Genes are **clipped to their bounds** after crossover.

**Crossover probability:** 0.7 (70% of parent pairs undergo crossover)

### Why SBX for this problem?

SBX preserves the **continuous nature** of inclination and azimuth angles. It produces offspring that are typically near both parents but can occasionally be far away, balancing exploitation and exploration.

---

## 8. Mutation — Polynomial Bounded Mutation

A **polynomial bounded mutation** (`tools.mutPolynomialBounded`) is applied to each gene with probability $p_m = 0.2$.

The mutation perturbation for gene $k$:

$$\delta_q = \begin{cases}
(2u + (1-2u)(1-\delta_1)^{\eta_m+1})^{1/(\eta_m+1)} - 1 & \text{if } u < 0.5 \\
1 - (2(1-u) + 2(u-0.5)(1-\delta_2)^{\eta_m+1})^{1/(\eta_m+1)} & \text{if } u \geq 0.5
\end{cases}$$

where $\eta_m = 20$ (distribution index) and $\delta_1, \delta_2$ are the fractional distances to the lower and upper bounds.

**Effect:** Small, bounded perturbations that explore the neighbourhood of the current solution without violating angle constraints.

**Mutation probability per gene:** 0.2 — roughly 1 in 5 genes mutates per individual.

---

## 9. Generational Loop

```
Initialise population (N=50 individuals)
Evaluate all individuals → assign fitness scores

For generation g = 1 to G (default G=100):
    1. Tournament selection → mating pool (N individuals)
    2. Clone mating pool → offspring
    3. Pairwise SBX crossover (prob=0.7)
    4. Per-gene polynomial mutation (prob=0.2 per gene)
    5. Re-evaluate only modified individuals (fitness invalid)
    6. Replace population with offspring (generational GA)
    7. Record best fitness → generation_history[g]

Return: best individual, decoded trajectory, generation_history
```

**Elitism:** WellPath.AI uses the DEAP `eaSimple` style (offspring fully replace population). The best individual is tracked via `generation_history` for the convergence chart.

## 10. Convergence Analysis

The **generation_history** array records the best fitness at each generation. A typical convergence curve shows three phases:

```
Fitness
  1.0 |                                          ████████████
      |                              ████████████
  0.6 |                  ████████████
      |      ████████████
  0.3 |██████
      |
  0.0 +—————————————————————————————————————————————→ Generation
       0    20    40    60    80    100
       ↑                   ↑            ↑
    Random start     Rapid improvement  Plateau (converged)
```

| Phase | Description |
|---|---|
| **Exploration** (gen 1–20) | Population diversity high; fitness varies widely |
| **Exploitation** (gen 20–70) | Good solutions selected; rapid improvement |
| **Convergence** (gen 70+) | Population homogeneous; marginal gains |

The GA convergence chart in WellPath.AI's 3D view shows this curve with a gradient fill, and reports the total fitness improvement (e.g., "+42.3%").

## 11. GA Parameters and Their Effect

| Parameter | Default | Effect of increasing |
|---|---|---|
| `population` | 50 | Better global coverage, slower per generation |
| `generations` | 100 | More iterations, better convergence (diminishing returns after ~150) |
| `waypoints` | 8 | More trajectory control points, more complex paths possible |
| `dls_weight` | 0.3 | Smoother path, potentially less productive zone contact |
| Crossover prob | 0.7 | More recombination, faster mixing of good genes |
| Mutation prob | 0.2 | More exploration, prevents premature convergence |
| `eta` (SBX/mutation) | 20 | Higher → offspring closer to parents (exploitation) |

## 12. Why Genetic Algorithm vs. Other Optimization Methods

| Method | Principle | Requires gradient? | Good for this problem? |
|---|---|---|---|
| **Gradient Descent** | Follow steepest descent | Yes | No — fitness not differentiable w.r.t. trajectory |
| **Newton's Method** | Quadratic approximation | Yes (Hessian) | No |
| **Simulated Annealing** | Random walk + cooling | No | Yes, but slower than GA for this dimensionality |
| **Particle Swarm Optimization (PSO)** | Swarm intelligence | No | Comparable to GA; less theoretical framework |
| **Bayesian Optimization** | Gaussian process surrogate | No | Good for expensive functions; overkill here |
| **Genetic Algorithm** ✓ | Evolutionary selection | No | Yes — handles constraints, non-linear, multi-modal |

**Key advantages of GA for well path optimization:**
1. **No gradient needed** — trajectory fitness is computed via MCM + interpolation, no closed form
2. **Global search** — population explores many regions simultaneously
3. **Constraint handling** — bounds enforced directly in initialization, crossover, mutation
4. **Interpretable evolution** — convergence history is a natural audit trail
5. **Industry precedent** — GAs have been applied to well path optimisation since the 1990s (Siddiqui et al., 2012)

## 13. DEAP Framework

WellPath.AI uses **DEAP** (Distributed Evolutionary Algorithms in Python) — a flexible, well-documented EA framework:

```python
from deap import base, creator, tools

# Define fitness (maximization) and individual types
creator.create("FitnessMax", base.Fitness, weights=(1.0,))
creator.create("Individual", list, fitness=creator.FitnessMax)

# Register operators
toolbox.register("evaluate", fitness_function)
toolbox.register("mate",    tools.cxSimulatedBinaryBounded, ...)
toolbox.register("mutate",  tools.mutPolynomialBounded, ...)
toolbox.register("select",  tools.selTournament, tournsize=3)
```

DEAP separates the EA logic from the problem definition, making it easy to swap operators without changing the optimization loop.

## 14. Post-Processing — Formation Layers for the 3D Scene

After the GA returns the best trajectory, the backend converts the per-depth productivity scores into a **layer model** that the 3D scene renders as colored slabs behind the trajectory. This is what gives the 3D view its productive (green), marginal (amber), and non-productive (red) bands.

### 14.1 Pipeline

```
productivity_scores  ──►  smooth (edge-padded moving average)
                          │
                          ▼
                       threshold
                          │
                          ├─► score ≥ 0.50  → productive
                          ├─► 0.25 ≤ s < 0.50 → marginal
                          └─► score <  0.25 → non-productive
                          │
                          ▼
                  group consecutive same-label depths
                       (raw_layers)
                          │
                          ▼
                  iteratively merge layers thinner than
                  min_layer_thickness = 30 m, preserving
                  the DOMINANT label across the merge
                          │
                          ▼
              formation_layers  ──► response payload
              [{depth_top, depth_bottom, label, avg_score}, ...]
```

### 14.2 Edge-padded smoothing

The moving-average kernel uses **`np.pad(scores, pad, mode="edge")`** before convolution rather than `np.convolve(..., mode="same")`. Implicit-zero padding biased the smoothed series toward zero at the start and end of the well, which silently relabelled productive bands near the surface or bottom as marginal/non-productive — they then disappeared from the 3D scene. Reflective edge padding eliminates this boundary artifact.

### 14.3 Dominant-label preservation during merging

When a thin layer (< 30 m) is merged into its neighbour, the merged layer takes the **highest-priority label** between the two:

```
priority = {productive: 2, marginal: 1, non-productive: 0}
merged.label = argmax priority(prev.label, layer.label)
```

A naïve merge would overwrite a thin productive sliver with the surrounding marginal/non-productive label, removing it from the final layer model. The dominant-label rule guarantees that *every* productive band detected by XGBoost survives the merge step and appears in the 3D scene as a green slab — even when that band is sandwiched between thicker non-productive zones.

### 14.4 Why this matters for the GA

The GA does **not** use `formation_layers` directly — its fitness function operates on the raw per-depth `productivity_scores`. But the layer model is the user-facing artifact the supervisor sees in the 3D scene. If a productive zone is missing from the slabs, the optimizer's behaviour looks confusing ("why is the path bending here?") even when the underlying GA is doing the right thing. The layer-construction algorithm exists so the visualisation matches the optimisation target.

## 15. Stochasticity and Reproducibility

**GAs are stochastic** by design — two runs with different random seeds will produce different (but usually similarly fit) solutions. This is a **feature, not a bug**:

- Different runs explore different parts of the solution space
- The best result can be obtained by running multiple times and taking the best
- For thesis reproducibility, the synthetic data seed is fixed (seed=2007007), but GA randomness is preserved to demonstrate the algorithm's evolutionary nature

**Defense answer for "why do I get different paths?":**
> "The Genetic Algorithm is inherently stochastic — like natural evolution, it uses randomness to explore the solution space. With the same synthetic well log data, predictions are now deterministic (fixed seed), but the GA explores a different random trajectory through the solution space each run. This is the correct behaviour — it demonstrates the algorithm is genuinely searching, not following a deterministic script. In practice, you run the GA multiple times and keep the best solution."
